# Memo-Pad 开发规范 v1.0

## 核心开发原则（不可变）

1. **企业级对标** — 每次开发前检索 ≥2 个成熟产品/案例作为参照
2. **一次规划、不再回头** — 评审门禁通过后，除非遇到不可预见的阻塞，否则不修改方案
3. **渐进式验证** — 每完成一个文件立即编译检查，不等最后集中修复
4. **零返工设计** — 修改前先读代码理解上下文，变更仅限目标范围
5. **Token 效率优先** — 批量并行读文件、合并小修改、删除冗余步骤

## 禁止事项

- **禁止在 Phase 5 门禁通过后推翻方案**
- **禁止未经阅读代码就修改文件** — 所有修改前必须先 Read
- **禁止一次性修改 >5 个文件不编译验证**
- **禁止添加不必要的代码注释** — 保持代码自解释
- **禁止循环讨论已确认的决策**

## 必须事项

- **每个文件修改前必须 Read 读取**
- **每次修改后必须编译/类型检查验证**
- **使用批量并行工具调用减少轮次**
- **代码风格严格跟随现有文件模式** — 不引入新框架/风格
- **每模块完成后 Git commit**

## 7 阶段开发流程（带硬性门禁）

### Phase 1: 需求分析
- 1.1 明确功能边界（用户故事 → 验收条件）
- 1.2 可行性判断（技术栈限制、资源评估）
- 输出：功能需求清单（≤10 条）

### Phase 2: 对标研究
- 2.1 检索 ≥2 个行业最佳案例/开源项目
- 2.2 提取可复用的架构模式和设计决策
- 输出：对标分析摘要 + 技术方案草案

### Phase 3: 技术设计
- 3.1 技术栈选定（优先复用现有依赖）
- 3.2 接口契约定义（REST API / 类接口签名）
- 3.3 文件级变更清单（新增/修改/删除）
- 输出：完整设计文档

### Phase 4: 风险评审
- 4.1 识别 ≥3 个潜在风险 + 缓解措施
- 4.2 评估对现有功能的影响范围
- 输出：风险评估矩阵

### ★★★ Phase 5: 评审门禁（HARD GATE）★★★
- 5.1 用户确认计划 ← **关键决策点**
- 5.2 门禁通过后方案冻结，禁止架构级修改
- 输出：用户签字的最终方案

### Phase 6: 增量开发
- 6.1 按文件逐个实现（每文件：读→改→编译验证）
- 6.2 模块级自测（每 3-5 个文件完成一个子功能）
- 6.3 Git 增量提交（每模块一个 commit）
- 输出：可用代码 + 编译通过

### Phase 7: 验证交付
- 7.1 全量编译 + Lint 检查
- 7.2 边界场景测试（空输入/超长/并发/注入）
- 7.3 变更总结报告
- 输出：测试结果 + 变更清单
