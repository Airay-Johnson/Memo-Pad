# Memo-Pad 优化迭代流程 v1.1

## Phase 0: 问题确认（新增★）

**任何优化/修复必须先执行此阶段，禁止跳过。**

1. **复现确认**：读取相关代码，确认问题是否可复现
2. **根因定位**：逐层追查（前端→网络→后端→数据库），找到真正的问题代码
3. **架构审查**：检查是否属于架构级缺陷（如数据模型、调用链路设计），而非局部 bug
4. **影响评估**：列出受影响的文件/功能范围
5. **方案确认**：向用户描述根因 + 影响范围，获得确认后再制定修改计划

> **严禁**：在未确认根因前直接修改代码。严禁在出现数据权限/隔离问题时只修复前端缓存。

## 核心原则

1. **单点改动** — 每次只处理一个优化点，验证后再继续
2. **双验证** — 改前基准测量 + 改后对比验证
3. **可回滚** — 每次改动保持独立，可单步回退
4. **优先级驱动** — 按影响/成本比排序，先做高收益低风险

## 3 阶段优化循环

### Phase 1: 诊断分析
- **如果是 Bug**：复现步骤 → 根因定位 → 影响范围评估
- **如果是性能**：profiling 定位瓶颈 → 量化当前指标 → 设定优化目标
- **如果是重构**：识别坏味道 → 评估改造范围 → 对比行业标准
- **输出**：诊断报告(含基准数据)

### Phase 2: 定点优化
- 只改目标代码，不碰无关部分
- 小步快跑：修改 → 编译 → 验证，每步确认
- 保留向后兼容（除非明确要求 breaking change）
- **输出**：改动代码 + 编译通过

### Phase 3: 回归验证
- 改前改后指标对比(性能/代码行数/复杂度)
- 运行现有测试套件确认无回归
- 如涉及行为变更，补充测试用例
- **输出**：对比报告 + 结论

## 代码审查清单

修改时逐项检查：
- [ ] 是否引入新依赖？（优先复用现有）
- [ ] 是否破坏现有 API 签名？
- [ ] 是否改变了异常处理语义？
- [ ] 是否有未处理的边界条件？
- [ ] 是否与现有代码风格一致？
- [ ] 是否保留了安全护栏逻辑？

## 优化优先级矩阵

```
影响大  │ P1: 立即修复         │ P2: 规划迭代
        │ (崩溃/安全/数据错)    │ (性能瓶颈/代码腐烂)
        │                      │
影响小  │ P4: 暂缓             │ P3: 顺手改进
        │ (锦上添花)            │ (命名/注释/格式)
        └────────────────────────────────
          成本低               成本高
```

## 性能优化指南

### Java 后端
- 检查点：N+1 查询、数据库连接池、线程池大小
- AI 调用：Token 消耗、API 超时、重试策略
- 内存：EmbeddingStore 大小、ChatMemory 窗口配置

### Vue 前端
- 组件渲染：大列表用虚拟滚动、避免不必要的 re-render
- Bundle 大小：按需加载对话框组件、图片懒加载
- 网络：流式输出断线重连、请求去重

## 回滚策略

每次优化做一个独立 commit，标题格式：
```
<type>(<scope>): <description>

type: perf | fix | refactor | style
scope: agent | controller | service | frontend
```

回退命令：`git revert <commit-hash>`
